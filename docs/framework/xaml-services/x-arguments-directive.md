# <a name="xarguments-directive"></a>Директива x:Arguments
Упаковывает аргументы создания для объявления элемент конструктора не по умолчанию объект в XAML или объект объявления метода фабрики.  
  
## <a name="xaml-element-usage-nondefault-constructor"></a>Использование элементов XAML (конструктора не по умолчанию)  
  
```  
<object ...>  
  <x:Arguments>  
    oneOrMoreObjectElements  
  </x:Arguments>  
</object>  
```  
  
## <a name="xaml-element-usage-factory-method"></a>Использование элементов XAML (фабричный метод)  
  
```  
<object x:FactoryMethod="methodName"...>  
  <x:Arguments>  
    oneOrMoreObjectElements  
  </x:Arguments>  
</object>  
```  
  
## <a name="xaml-values"></a>Значения XAML  
  
|||  
|-|-|  
|`oneOrMoreObjectElements`|Один или несколько элементов объектов, указывающих аргументы, передаваемые методом фабрики или конструктора не по умолчанию резервным.<br /><br /> Типичное применение — использовать текст инициализации в элементы объекта значения фактического аргумента. Примеры см. раздел.<br /><br /> Порядок элементов имеет значение. Типов XAML в порядке должен совпадать с типами и введите порядок резервного конструктора или фабрики перегрузки метода.|  
|`methodName`|Имя фабричный метод, который должен обрабатывать любые `x:Arguments` аргументы.|  
  
## <a name="dependencies"></a>Зависимости  
 `x:FactoryMethod` область действия и поведение можно изменить где `x:Arguments` применяется.  
  
 Если не `x:FactoryMethod` указано, `x:Arguments` применяется к альтернативные подписи (а не стандартной) резервных конструкторов.  
  
 Если `x:FactoryMethod` указано, `x:Arguments` применяется к перегрузку именованного метода.  
  
## <a name="remarks"></a>Примечания  
 Поддерживает инициализации не по умолчанию через текст инициализации XAML 2006. Тем не менее Практическое применение методики построения текста инициализации ограничено. Текст инициализации рассматривается как единственная текстовая строка; Таким образом он только добавляет возможность инициализации одного параметра, если преобразователь типов, определенных для поведение конструктора, который может выполнять синтаксический анализ сведений о настраиваемом элементов и пользовательских разделителей в строке. Кроме того текстовая строка для логики объекта является потенциально заданного анализатор XAML собственного умолчанию преобразователь типов для примитивов, отличное от true строки обработки.  
  
 `x:Arguments` Использование XAML не использование элемента свойства, в том смысле, типичный, так как директив разметки не ссылается на объект элемента, содержащего типа. Это больше похоже на другие директивы, такие как `x:Code` где этот элемент разграничивает диапазон, в котором разметку следует интерпретировать как и в случае кроме порта по умолчанию для дочернего содержимого. В этом случае тип каждого элемента объекта XAML передает сведения о типы аргументов, которая используется средства синтаксического анализа XAML, чтобы определить, какие сигнатуру метода фабрики определенный конструктор `x:Arguments` использования попытка ссылки.  
  
 `x:Arguments` для объектного элемента, создаваемого должен находиться перед других свойств элементов, содержимое, внутренний текст и строки инициализации объектного элемента. Объектные элементы в пределах `x:Arguments` могут включать атрибуты и строки инициализации с этого типа XAML и его базовым методом фабрики или конструктора. Для объекта или аргументы можно указать пользовательские типы XAML или XAML, в противном случае за пределами пространства имен XAML по умолчанию, ссылаясь на установленный префикс сопоставления.  
  
 Процессоры XAML следуйте приведенным ниже рекомендациям, чтобы определить, как указано аргументы в `x:Arguments` следует использовать для создания объекта. Если `x:FactoryMethod` указан, сведения сравниваются в указанный `x:FactoryMethod` (Обратите внимание, что значение `x:FactoryMethod` — это имя метода, а именованный метод может иметь перегрузки. Если `x:FactoryMethod` не указан, сведения сравниваются в набор для всех перегрузок открытый конструктор объекта. Логика обработки XAML затем сравнивает число параметров и выбирает перегрузка с соответствующим арности. Если имеется более одного совпадения, процессор XAML должен сравнить типы параметров, исходя из типов XAML, предоставленный объект элементов. Есть ли по-прежнему более одного совпадения, поведение обработчика XAML не определено. Если `x:FactoryMethod` указан, но метод не может быть разрешена, процессор XAML должен создать исключение.  
  
 Использование атрибута XAML `<x:Arguments>string</x:Arguments>` технически возможно. Тем не менее это обеспечивает нет новых возможностей, что можно сделать в противном случае через инициализации текста и преобразователи типов, и с помощью этого синтаксиса не является необходимым в возможности XAML 2009 г. метод фабрики.  
  
## <a name="examples"></a>Примеры  
 В следующем примере показано конструктора не по умолчанию подпись, а затем использование XAML `x:Arguments` , обращающийся к подписи.  
  
```csharp  
public class Food {  
    private string _name;  
    private Int32 _calories;  
    public Food(string name, Int32 calories) {  
        _name=name;  
        _calories=calories;  
    }  
}  
```  
  
```xaml  
<my:Food>  
    <x:Arguments>  
        <x:String>Apple</x:String>  
        <x:Int32>150</x:Int32>  
    </x:Arguments>  
</my:Food>  
```  
  
 В следующем примере показано сигнатуры метода на целевой объект фабрики, а затем использование XAML `x:Arguments` , обращающийся к подписи.  
  
```csharp  
public Food TryLookupFood(string name)  
{  
  switch (name) {  
    case "Apple": return new Food("Apple",150);  
    case "Chocolate": return new Food("Chocolate",200);  
    case "Cheese": return new Food("Cheese", 450);  
    default: {return new Food(name,0);  
  }  
}  
```  
  
```xaml  
<my:Food x:FactoryMethod="TryLookupFood">  
    <x:Arguments>  
        <x:String>Apple</x:String>  
    </x:Arguments>  
</my:Food>  
```
