# <a name="creating-application-roles-in-sql-server"></a><span data-ttu-id="b89aa-102">Создание ролей приложений в SQL Server</span><span class="sxs-lookup"><span data-stu-id="b89aa-102">Creating Application Roles in SQL Server</span></span>
<span data-ttu-id="b89aa-103">Роли приложения предоставляют возможность назначать разрешения приложению, а не ролям базы данных или пользователям.</span><span class="sxs-lookup"><span data-stu-id="b89aa-103">Application roles provide a way to assign permissions to an application instead of a database role or user.</span></span> <span data-ttu-id="b89aa-104">Пользователи могут соединиться с базой данных, активировать роль приложения и воспользоваться разрешениями, предоставленными приложению.</span><span class="sxs-lookup"><span data-stu-id="b89aa-104">Users can connect to the database, activate the application role, and assume the permissions granted to the application.</span></span> <span data-ttu-id="b89aa-105">Разрешения, предоставленные роли приложения, действуют в течение всего соединения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-105">The permissions granted to the application role are in force for the duration of the connection.</span></span>  
  
> [!IMPORTANT]
> <span data-ttu-id="b89aa-106">Роли приложения активируются, когда клиентское приложение передает имя роли приложения и пароль в строке соединения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-106">Application roles are activated when a client application supplies an application role name and a password in the connection string.</span></span> <span data-ttu-id="b89aa-107">В двухуровневом приложении безопасность при этом снижается, поскольку пароль должен храниться на клиентском компьютере.</span><span class="sxs-lookup"><span data-stu-id="b89aa-107">They present a security vulnerability in a two-tier application because the password must be stored on the client computer.</span></span> <span data-ttu-id="b89aa-108">В трехуровневом приложении пароль можно хранить так, чтобы он не был доступен пользователям приложения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-108">In a three-tier application, you can store the password so that it cannot be accessed by users of the application.</span></span>  
  
## <a name="application-role-features"></a><span data-ttu-id="b89aa-109">Функции ролей приложения</span><span class="sxs-lookup"><span data-stu-id="b89aa-109">Application Role Features</span></span>  
 <span data-ttu-id="b89aa-110">Роли приложения имеют следующие возможности.</span><span class="sxs-lookup"><span data-stu-id="b89aa-110">Application roles have the following features:</span></span>  
  
- <span data-ttu-id="b89aa-111">В отличие от ролей базы данных роли приложения не содержат элементов.</span><span class="sxs-lookup"><span data-stu-id="b89aa-111">Unlike database roles, application roles contain no members.</span></span>  
  
- <span data-ttu-id="b89aa-112">Роли приложения активируются, когда клиентское приложение передает имя и пароль роли приложения системной хранимой процедуре `sp_setapprole`.</span><span class="sxs-lookup"><span data-stu-id="b89aa-112">Application roles are activated when an application supplies the application role name and a password to the `sp_setapprole` system stored procedure.</span></span>  
  
- <span data-ttu-id="b89aa-113">Пароль должен храниться на клиентском компьютере и передаваться во время выполнения. Роль приложения нельзя активировать из SQL Server.</span><span class="sxs-lookup"><span data-stu-id="b89aa-113">The password must be stored on the client computer and supplied at run time; an application role cannot be activated from inside of SQL Server.</span></span>  
  
- <span data-ttu-id="b89aa-114">Пароль не зашифрован.</span><span class="sxs-lookup"><span data-stu-id="b89aa-114">The password is not encrypted.</span></span> <span data-ttu-id="b89aa-115">При сохранении пароля используется одностороннее хэширование.</span><span class="sxs-lookup"><span data-stu-id="b89aa-115">The parameter password is stored as a one-way hash.</span></span>  
  
- <span data-ttu-id="b89aa-116">После активации разрешения, полученные через роль приложения, действуют в течение всего соединения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-116">Once activated, permissions acquired through the application role remain in effect for the duration of the connection.</span></span>  
  
- <span data-ttu-id="b89aa-117">Роль приложения наследует разрешения, предоставленные роли `public`.</span><span class="sxs-lookup"><span data-stu-id="b89aa-117">The application role inherits permissions granted to the `public` role.</span></span>  
  
- <span data-ttu-id="b89aa-118">Если роль приложения активирует член предопределенной роли сервера `sysadmin`, контекст безопасности переключается на контекст безопасности роли приложения на все время соединения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-118">If a member of the `sysadmin` fixed server role activates an application role, the security context switches to that of the application role for the duration of the connection.</span></span>  
  
- <span data-ttu-id="b89aa-119">Если создать учетную запись `guest` в базе данных, в которой имеется роль приложения, то не нужно будет создавать учетную запись для роли приложения или для любого имени входа, вызывающего эту роль.</span><span class="sxs-lookup"><span data-stu-id="b89aa-119">If you create a `guest` account in a database that has an application role, you do not need to create a database user account for the application role or for any of the logins that invoke it.</span></span> <span data-ttu-id="b89aa-120">Роли приложения могут получать прямой доступ к другой базе данных, только если во второй базе данных существует учетная запись `guest`.</span><span class="sxs-lookup"><span data-stu-id="b89aa-120">Application roles can directly access another database only if a `guest` account exists in the second database</span></span>  
  
- <span data-ttu-id="b89aa-121">Встроенные функции, возвращающие имена входа, например SYSTEM_USER, возвращают имя входа, которое вызвало роль приложения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-121">Built-in functions that return login names, such as SYSTEM_USER, return the name of the login that invoked the application role.</span></span> <span data-ttu-id="b89aa-122">Встроенные функции, возвращающие имена пользователей базы данных, возвращают имя роли приложения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-122">Built-in functions that return database user names return the name of the application role.</span></span>  
  
### <a name="the-principle-of-least-privilege"></a><span data-ttu-id="b89aa-123">Принцип минимальных привилегий</span><span class="sxs-lookup"><span data-stu-id="b89aa-123">The Principle of Least Privilege</span></span>  
 <span data-ttu-id="b89aa-124">Ролям приложений следует предоставлять только необходимые разрешения на случай, если пароль будет раскрыт.</span><span class="sxs-lookup"><span data-stu-id="b89aa-124">Application roles should be granted only required permissions in case the password is compromised.</span></span> <span data-ttu-id="b89aa-125">Разрешения роли `public` необходимо отменить во всех базах данных, использующих роль приложения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-125">Permissions to the `public` role should be revoked in any database using an application role.</span></span> <span data-ttu-id="b89aa-126">Отключите учетную запись `guest` во всех базах данных, к которым не следует предоставлять доступ участникам роли приложения.</span><span class="sxs-lookup"><span data-stu-id="b89aa-126">Disable the `guest` account in any database you do not want callers of the application role to have access to.</span></span>  
  
### <a name="application-role-enhancements"></a><span data-ttu-id="b89aa-127">Расширения роли приложения</span><span class="sxs-lookup"><span data-stu-id="b89aa-127">Application Role Enhancements</span></span>  
 <span data-ttu-id="b89aa-128">Контекст выполнения после активации роли приложения можно переключить обратно на исходного участника, устранив тем самым необходимость отключения пула соединений.</span><span class="sxs-lookup"><span data-stu-id="b89aa-128">The execution context can be switched back to the original caller after activating an application role, removing the need to disable connection pooling.</span></span> <span data-ttu-id="b89aa-129">Хранимая процедура `sp_setapprole` получила новый параметр, создающий файл cookie, который содержит контекстные сведения об участнике.</span><span class="sxs-lookup"><span data-stu-id="b89aa-129">The `sp_setapprole` procedure has a new option that creates a cookie, which contains context information about the caller.</span></span> <span data-ttu-id="b89aa-130">Сеанс можно восстановить, вызвав процедуру `sp_unsetapprole` и передав ей этот файл cookie.</span><span class="sxs-lookup"><span data-stu-id="b89aa-130">You can revert the session by calling the `sp_unsetapprole` procedure, passing it the cookie.</span></span>
