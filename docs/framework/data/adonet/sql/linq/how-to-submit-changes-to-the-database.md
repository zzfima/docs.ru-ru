# <a name="how-to-submit-changes-to-the-database"></a><span data-ttu-id="a17bd-102">Как выполнить отправить изменения в базу данных</span><span class="sxs-lookup"><span data-stu-id="a17bd-102">How to: Submit Changes to the Database</span></span>
<span data-ttu-id="a17bd-103">Независимо от количества изменений, произведенных над объектами, эти изменения выполняются только над репликам, содержащимся в памяти.</span><span class="sxs-lookup"><span data-stu-id="a17bd-103">Regardless of how many changes you make to your objects, changes are made only to in-memory replicas.</span></span> <span data-ttu-id="a17bd-104">Фактические данные в базе данных при этом не изменяются.</span><span class="sxs-lookup"><span data-stu-id="a17bd-104">You have made no changes to the actual data in the database.</span></span> <span data-ttu-id="a17bd-105">Изменения не передаются на сервер до тех пор, пока не будет явно вызван метод <xref:System.Data.Linq.DataContext.SubmitChanges%2A> класса <xref:System.Data.Linq.DataContext>.</span><span class="sxs-lookup"><span data-stu-id="a17bd-105">Your changes are not transmitted to the server until you explicitly call <xref:System.Data.Linq.DataContext.SubmitChanges%2A> on the <xref:System.Data.Linq.DataContext>.</span></span>  
  
 <span data-ttu-id="a17bd-106">При вызове этого метода класс <xref:System.Data.Linq.DataContext> пытается преобразовать сделанные изменения в эквивалентные команды SQL.</span><span class="sxs-lookup"><span data-stu-id="a17bd-106">When you make this call, the <xref:System.Data.Linq.DataContext> tries to translate your changes into equivalent SQL commands.</span></span> <span data-ttu-id="a17bd-107">Можно использовать собственную пользовательскую логику для переопределения этих действий, однако порядок отправки управляется службой для <xref:System.Data.Linq.DataContext> называется *обработчиком изменений*.</span><span class="sxs-lookup"><span data-stu-id="a17bd-107">You can use your own custom logic to override these actions, but the order of submission is orchestrated by a service of the <xref:System.Data.Linq.DataContext> known as the *change processor*.</span></span> <span data-ttu-id="a17bd-108">Ниже представлена последовательность событий.</span><span class="sxs-lookup"><span data-stu-id="a17bd-108">The sequence of events is as follows:</span></span>  
  
1.  <span data-ttu-id="a17bd-109">При вызове метода <xref:System.Data.Linq.DataContext.SubmitChanges%2A> технология [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] просматривает набор известных объектов, чтобы определить, присоединены ли к ним новые экземпляры.</span><span class="sxs-lookup"><span data-stu-id="a17bd-109">When you call <xref:System.Data.Linq.DataContext.SubmitChanges%2A>, [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] examines the set of known objects to determine whether new instances have been attached to them.</span></span> <span data-ttu-id="a17bd-110">При положительном результате новые экземпляры добавляются в набор отслеживаемых объектов.</span><span class="sxs-lookup"><span data-stu-id="a17bd-110">If they have, these new instances are added to the set of tracked objects.</span></span>  
  
2.  <span data-ttu-id="a17bd-111">Все объекты, содержащие ожидающие изменения, упорядочиваются в последовательности объектов на основе зависимостей между ними.</span><span class="sxs-lookup"><span data-stu-id="a17bd-111">All objects that have pending changes are ordered into a sequence of objects based on the dependencies between them.</span></span> <span data-ttu-id="a17bd-112">Объекты, изменения которых зависят от других объектов, располагаются после своих зависимостей.</span><span class="sxs-lookup"><span data-stu-id="a17bd-112">Objects whose changes depend on other objects are sequenced after their dependencies.</span></span>  
  
3.  <span data-ttu-id="a17bd-113">Непосредственно после передачи фактических изменений [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] начинает транзакцию для инкапсуляции ряда отдельных команд.</span><span class="sxs-lookup"><span data-stu-id="a17bd-113">Immediately before any actual changes are transmitted, [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] starts a transaction to encapsulate the series of individual commands.</span></span>  
  
4.  <span data-ttu-id="a17bd-114">Изменения объектов последовательно преобразуются в команды SQL и отправляются на сервер.</span><span class="sxs-lookup"><span data-stu-id="a17bd-114">The changes to the objects are translated one by one to SQL commands and sent to the server.</span></span>  
  
 <span data-ttu-id="a17bd-115">В данный момент любые ошибки, обнаруженные базой данных, приводят к остановке процесса отправки и вызову исключения.</span><span class="sxs-lookup"><span data-stu-id="a17bd-115">At this point, any errors detected by the database cause the submission process to stop, and an exception is raised.</span></span> <span data-ttu-id="a17bd-116">Выполняется откат всех изменений базы данных к состоянию до начала отправки.</span><span class="sxs-lookup"><span data-stu-id="a17bd-116">All changes to the database are rolled back as if no submissions ever occurred.</span></span> <span data-ttu-id="a17bd-117">Класс <xref:System.Data.Linq.DataContext> по-прежнему содержит запись всех изменений.</span><span class="sxs-lookup"><span data-stu-id="a17bd-117">The <xref:System.Data.Linq.DataContext> still has a full recording of all changes.</span></span> <span data-ttu-id="a17bd-118">Поэтому можно попытаться исправить проблему и вызвать метод <xref:System.Data.Linq.DataContext.SubmitChanges%2A> еще раз, как показано в следующем примере кода.</span><span class="sxs-lookup"><span data-stu-id="a17bd-118">You can therefore try to correct the problem and call <xref:System.Data.Linq.DataContext.SubmitChanges%2A> again, as in the code example that follows.</span></span>
