<span data-ttu-id="3f0d0-105">Представьте себе контейнер как экземпляр процесса.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-105">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="3f0d0-106">Процесс не находится в неизменном состоянии.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-106">A process doesn't maintain persistent state.</span></span> <span data-ttu-id="3f0d0-107">И хотя контейнер может записывать данные в локальное хранилище, предполагать, что экземпляр всегда будет на месте — это все равно, что надеяться на постоянство одной ячейки памяти.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-107">While a container can write to its local storage, assuming that an instance will be around indefinitely would be like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="3f0d0-108">Вы должны предполагать, что образы контейнеров, как и процессы, имеют несколько экземпляров или конечном итоге будут удалены.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-108">You should assume that container images, like processes, have multiple instances or will eventually be killed.</span></span> <span data-ttu-id="3f0d0-109">Если для управления ими используется оркестратор контейнеров, следует предполагать, что они могут перемещаться из одного узла (или виртуальной машины) в другой.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-109">If they're managed with a container orchestrator, you should assume that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="3f0d0-110">Для управления данными в приложениях Docker используются следующие решения:</span><span class="sxs-lookup"><span data-stu-id="3f0d0-110">The following solutions are used to manage data in Docker applications:</span></span>

<span data-ttu-id="3f0d0-111">Из узла Docker в качестве [томов Docker](https://docs.docker.com/engine/admin/volumes/):</span><span class="sxs-lookup"><span data-stu-id="3f0d0-111">From the Docker host, as [Docker Volumes](https://docs.docker.com/engine/admin/volumes/):</span></span>

- <span data-ttu-id="3f0d0-112">**Тома** хранятся в той части файловой системы узла, которая находится под управлением Docker.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-112">**Volumes** are stored in an area of the host filesystem that's managed by Docker.</span></span>

- <span data-ttu-id="3f0d0-113">**Подключения привязок** можно сопоставить с любой папкой в файловой системе узла, поэтому доступом нельзя управлять из процесса Docker, что может представлять угрозу для безопасности, поскольку контейнер может получить доступ к конфиденциальным папкам операционной системы.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-113">**Bind mounts** can map to any folder in the host filesystem, so access can't be controlled from Docker process and can pose a security risk as a container could access sensitive OS folders.</span></span>

- <span data-ttu-id="3f0d0-114">**Подключения tmpfs** аналогичны виртуальным папкам, которые существуют только в памяти узла и никогда не записываются в файловую систему.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-114">**tmpfs mounts** are like virtual folders that only exist in the host's memory and are never written to the filesystem.</span></span>

<span data-ttu-id="3f0d0-115">Из удаленного хранилища:</span><span class="sxs-lookup"><span data-stu-id="3f0d0-115">From remote storage:</span></span>

- <span data-ttu-id="3f0d0-116">[Служба хранилища Azure](https://azure.microsoft.com/documentation/services/storage/), предоставляющая геораспределенное хранилище для долгосрочного хранения данных контейнеров.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-116">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/), which provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

- <span data-ttu-id="3f0d0-117">Удаленные реляционные базы данных, например [базы данных SQL Azure](https://azure.microsoft.com/services/sql-database/) или базы данных NoSQL, такие как [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), или службы кэша, такие как [Redis](https://redis.io/).</span><span class="sxs-lookup"><span data-stu-id="3f0d0-117">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/) or NoSQL databases like [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="3f0d0-118">Из контейнера Docker:</span><span class="sxs-lookup"><span data-stu-id="3f0d0-118">From the Docker container:</span></span>

- <span data-ttu-id="3f0d0-119">**Оверлейная файловая система**.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-119">**Overlay File System**.</span></span> <span data-ttu-id="3f0d0-120">Эта функция Docker выполняет копирование при записи, при котором обновленная информация сохраняется в корневой файловой системе контейнера.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-120">This Docker feature implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="3f0d0-121">Эта информация "накладывается" на изначальный образ, на котором базируется контейнер.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-121">That information is "on top" of the original image on which the container is based.</span></span> <span data-ttu-id="3f0d0-122">Если удалить контейнер из системы, эти изменения будут утеряны.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-122">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="3f0d0-123">Поэтому, хоть и возможно сохранить состояние контейнера в его локальном хранилище, создание системы по такой модели будет противоречить конструкции контейнера, состояние которого по умолчанию не отслеживается.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-123">Therefore, while it's possible to save the state of a container within its local storage, designing a system around this would conflict with the premise of container design, which by default is stateless.</span></span>

<span data-ttu-id="3f0d0-124">Тем не менее сейчас тома Docker являются предпочтительным способом обработки локальных данных в Docker.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-124">However, using Docker Volumes is now the preferred way to handle local data in Docker.</span></span> <span data-ttu-id="3f0d0-125">Если вам требуются дополнительные сведения о хранении в контейнерах, ознакомьтесь с разделами [Docker storage drivers](https://docs.docker.com/storage/storagedriver/select-storage-driver/) (Драйверы хранилища Docker) и [About storage drivers](https://docs.docker.com/storage/storagedriver/) (Основные сведения о драйверах хранилища).</span><span class="sxs-lookup"><span data-stu-id="3f0d0-125">If you need more information about storage in containers check on [Docker storage drivers](https://docs.docker.com/storage/storagedriver/select-storage-driver/) and [About storage drivers](https://docs.docker.com/storage/storagedriver/).</span></span>

<span data-ttu-id="3f0d0-126">Ниже подробно описаны эти варианты.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-126">The following provides more detail about these options:</span></span>

<span data-ttu-id="3f0d0-127">**Тома** — это каталоги из ОС узла, сопоставленные с каталогами в контейнерах.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-127">**Volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="3f0d0-128">Если код в контейнере имеет доступ к каталогу, на самом деле он обращается к каталогу на ОС узла.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-128">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="3f0d0-129">Этот каталог не привязан к времени существования самого контейнера, находится под управлением Docker и изолирован от основных функциональных возможностей хост-компьютера.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-129">This directory is not tied to the lifetime of the container itself, and the directory is managed by Docker and isolated from the core functionality of the host machine.</span></span> <span data-ttu-id="3f0d0-130">Поэтому тома данных хранят данные независимо от контейнера.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-130">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="3f0d0-131">Если удалить контейнер или образ из узла Docker, данные из томов данных не будут удалены.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-131">If you delete a container or an image from the Docker host, the data persisted in the data volume isn't deleted.</span></span>

<span data-ttu-id="3f0d0-132">Тома могут быть именованными или анонимными (по умолчанию).</span><span class="sxs-lookup"><span data-stu-id="3f0d0-132">Volumes can be named or anonymous (the default).</span></span> <span data-ttu-id="3f0d0-133">Именованные тома — это следующий этап развития **контейнеров томов данных**. Они упрощают обмен данными между контейнерами.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-133">Named volumes are the evolution of **Data Volume Containers** and make it easy to share data between containers.</span></span> <span data-ttu-id="3f0d0-134">Тома также поддерживают драйверы томов, позволяющие хранить данные на удаленных узлах и т. д.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-134">Volumes also support volume drivers that allow you to store data on remote hosts, among other options.</span></span>

<span data-ttu-id="3f0d0-135">**Подключения привязки** доступны довольно давно и позволяют сопоставлять любую папку с точкой подключения в контейнере.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-135">**Bind mounts** are available since a long time ago and allow the mapping of any folder to a mount point in a container.</span></span> <span data-ttu-id="3f0d0-136">Подключения привязки имеют больше ограничений, чем тома, и с ними связаны некоторые проблемы безопасности, поэтому тома являются рекомендуемым способом.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-136">Bind mounts have more limitations than volumes and some important security issues, so volumes are the recommended option.</span></span>

<span data-ttu-id="3f0d0-137">**Подключения tmpfs** по сути являются виртуальными папками, которые существуют только в памяти узла и никогда не записываются в файловую систему.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-137">**tmpfs mounts** are basically virtual folders that live only in the host's memory and are never written to the filesystem.</span></span> <span data-ttu-id="3f0d0-138">Они быстрые и безопасные, но используют ресурсы памяти и предназначены только для временных несохраняемых данных.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-138">They are fast and secure but use memory and are only meant for temporary, non-persistent data.</span></span>

<span data-ttu-id="3f0d0-139">Как показано на рисунке 4-5, обычные тома Docker могут храниться за пределами самих контейнерами, но в физических границах сервера узла или виртуальной машины.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-139">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="3f0d0-140">Тем не менее контейнеры Docker с одного сервера узла или виртуальной машины не могут обращаться к тому на другом сервере узла или виртуальной машине.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-140">However, Docker containers can't access a volume from one host server or VM to another.</span></span> <span data-ttu-id="3f0d0-141">Другими словами, с помощью этих томов невозможно управлять данными контейнеров, которые выполняются на разных узлах Docker, хотя такой режим работы можно реализовать с помощью драйвера томов, который поддерживает удаленные узлы.</span><span class="sxs-lookup"><span data-stu-id="3f0d0-141">In other words, with these volumes, it isn't possible to manage data shared between containers that run on different Docker hosts, although it could be achieved with a volume driver that supports remote hosts.</span></span>
