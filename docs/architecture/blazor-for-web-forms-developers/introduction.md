# <a name="an-introduction-to-blazor-for-aspnet-web-forms-developers"></a>Введение в Блазор для разработчиков веб-форм ASP.NET

[!INCLUDE [book-preview](../../../includes/book-preview.md)]

Платформа веб-форм ASP.NET была сшивания веб-разработки .NET с тех пор, как .NET Framework впервые поставлялась в 2002. Когда веб-узел по-прежнему находился в значительной сложности, ASP.NET веб-формы, создающие веб-приложения простыми и продуктивными, путем внедрения многих шаблонов, которые использовались для разработки настольных систем. В веб-формах ASP.NET веб-страницы могут быстро состоять из многократно используемых элементов управления пользовательского интерфейса. Взаимодействие с пользователем обрабатывается естественным образом как события. Существует обширная экосистема элементов управления пользовательского интерфейса веб-форм, предоставляемых корпорацией Майкрософт и разработчиками элементов управления. Элементы управления позволяют упростить подключение к источникам данных и отображение визуализаций с богатыми данными. Для визуально наклонного элемента конструктор веб-форм предоставляет простой интерфейс для управления элементами управления с помощью перетаскивания.

В течение многих лет корпорация Майкрософт представила новые веб-платформы на основе ASP.NET для решения тенденций веб-разработки. К некоторым таким веб-платформам относятся ASP.NET MVC, веб-страницы ASP.NET и недавно ASP.NET Core. С каждой новой платформой некоторые из них прогнозировать предполагаемый отказ от веб-форм ASP.NET и критиЦизед их как устаревшую, в режиме веб-платформы. Несмотря на эти прогнозы, многие веб-разработчики .NET продолжают находить ASP.NET Web Forms простым, стабильным и продуктивным способом выполнения своей работы.

На момент написания статьи почти половина веб-разработчиков использует веб-формы ASP.NET каждый месяц. Платформа веб-форм ASP.NET является стабильной до того момента, когда документы, образцы, книги и записи блога из десяти лет назад остаются полезными и актуальными. Для многих разработчиков веб-приложений .NET «ASP.NET» по-прежнему является синонимом «ASP.NET Web Forms», как это было при первом задумании .NET. Аргументы для достоинств и недостатков веб-форм ASP.NET по сравнению с другими новыми веб-платформами .NET могут быть выработаны. ASP.NET Web Forms остается популярной платформой для создания веб-приложений.

Даже поэтому инновации в разработке программного обеспечения не замедляют работу. Все разработчики программного обеспечения должны оставаться ориентироваться новых технологий и тенденций. В частности, следует учитывать две тенденции:

1. Переход к Open-Source и кросс-платформенному
2. Смена логики приложения на клиент

## <a name="an-open-source-and-cross-platform-net"></a>С открытым исходным кодом и между платформами .NET

При первой отгрузке веб-форм .NET и ASP.NET экосистема платформы выглядит значительно иначе, чем сегодня. Рынки для настольных компьютеров и серверов в Windows. Другие платформы, такие как macOS и Linux, по-прежнему испытывают трудности с обростом. ASP.NET Web Forms поставляется с .NET Framework в качестве компонента только для Windows, что означает, что ASP.NET приложения веб-форм могут работать только на компьютерах Windows Server. Многие современные среды теперь используют различные типы платформ для серверов и компьютеров разработки, так что межплатформенная поддержка для многих пользователей является абсолютной задачей.

Большинство современных веб-платформ теперь также имеют открытый исходный код, который имеет ряд преимуществ. Пользователи не принадлежат одному владельцу проекта для исправления ошибок и добавления компонентов. Проекты с открытым исходным кодом обеспечивают улучшенную прозрачность процесса разработки и предстоящие изменения. Проекты с открытым исходным кодом пользуются вкладом всего сообщества и способствуют поддержке экосистемы с открытым исходным кодом. Несмотря на риски с открытым исходным кодом, многие потребители и участники обнаружили подходящие меры по устранению, благодаря которым они смогут использовать преимущества экосистемы с открытым кодом в надежном и разумном виде. К таким примерам относятся лицензионные соглашения участника, удобные лицензии, педигриные проверки и вспомогательные решения.

Сообщество .NET использует как кросс-платформенную поддержку, так и открытый код. .NET Core — это межплатформенная реализация .NET, которая работает на разных платформах, включая Windows, macOS и различные дистрибутивы Linux. Xamarin предоставляет моно, версию .NET с открытым кодом. Моно работает на устройствах Android, iOS и различных других форм-факторах, включая контрольные значения и Интеллектуальные телевизоры. Корпорация Майкрософт объявила о том, что [.NET 5](https://devblogs.microsoft.com/dotnet/introducing-net-5/) будет согласовывать .NET Core и Mono с "единственной средой выполнения .NET и платформой, которую можно использовать везде и с единым поведением среды выполнения и опытом разработки".

Будет ли ASP.NET веб-форм преимущества от перехода на открытый исходный код и кросс-платформенная поддержка? Ответ, увы, — это нет или по крайней мере не в тот же экстент, что и остальная часть платформы. Недавно группа разработчиков .NET [стала ясно](https://devblogs.microsoft.com/dotnet/net-core-is-the-future-of-net/) , что веб-формы ASP.NET не будут перенесены в .NET Core или .NET 5. Почему так?

В ранних днях .NET Core для порта ASP.NET Web Forms были усилия. Обнаружено слишком радикальное количество критических изменений. Кроме того, здесь есть возможность допустить, что даже для корпорации Майкрософт существует ограничение числа веб-платформ, которые могут поддерживаться одновременно. Возможно, кто-то из сообщества выберет на себя создание веб-форм с открытым исходным кодом и кросс-платформенной версии ASP.NET. [Исходный код веб-форм ASP.NET](https://github.com/microsoft/referencesource) стал общедоступным в справочной форме. Но в то же время кажется, что ASP.NET веб-формы будут сохраняться только для Windows и без модели публикации с открытым исходным кодом. Если для ваших сценариев важна поддержка разных платформ или открытый исходный код, необходимо найти что-то новое.

Означает ли это, что ASP.NET веб-формы не *используются и больше* не должны использоваться? Конечно, нет! При условии, что .NET Framework поставляется в составе Windows, ASP.NET Web Forms будет поддерживаемой платформой. Для многих разработчиков веб-форм отсутствие различных платформ и поддержка открытых источников не вызывает проблем. Если у вас нет необходимости в использовании кросс-платформенной поддержки, с открытым исходным кодом или любых других новых функций в .NET Core или .NET 5, то при работе с ASP.NET Web Forms в Windows это нормально. ASP.NET Web Forms будет по-прежнему представлять собой эффективный способ написания веб-приложений на протяжении многих лет.

Но есть и другая тенденция, которую стоит рассмотреть, и это сдвиг к клиенту.

## <a name="client-side-web-development"></a>Веб-разработка на стороне клиента

Все. Веб-платформы на основе .NET, в том числе веб-формы ASP.NET, имеют один общий характер: они *визуализируются сервером*. В веб-приложениях, отображаемых сервером, браузер выполняет запрос к серверу, который выполняет некоторый код (код .NET в ASP.NET приложениях) для получения ответа. Этот ответ отправляется обратно в браузер для работы. В этой модели браузер используется в качестве механизма тонкой визуализации. Жесткая работа по созданию пользовательского интерфейса, запуску бизнес-логики и управлению состоянием происходит на сервере.

Однако браузеры стали универсальными платформами. Они реализуют постоянно увеличивающееся число открытых веб-стандартов, предоставляющих доступ к возможностям компьютера пользователя. Почему не следует использовать возможности вычислений, хранения, памяти и других ресурсов клиентского устройства? Взаимодействие пользовательского интерфейса в частности может воспользоваться преимуществами более богатого и интерактивного поведения при обработке по крайней мере частичной или полной клиентской стороны. Логика и данные, которые должны обрабатываться на сервере, по-прежнему могут обрабатываться на стороне сервера. Можно использовать вызовы веб-API или даже через протоколы реального времени, такие как WebSockets. Эти преимущества доступны разработчикам веб-приложений бесплатно, если они готовы писать JavaScript. Клиентские платформы пользовательского интерфейса, такие как угловые, реагирующие и Vue, упрощают веб-разработку на стороне клиента и увеличили популярность. Разработчики веб-форм ASP.NET также могут воспользоваться преимуществами клиента и даже иметь встроенную поддержку интегрированных платформ JavaScript, таких как ASP.NET AJAX.

Но за счет использования двух различных платформ и экосистем (.NET и JavaScript) поставляются затраты. Опыт необходим в двух параллельных средах с различными языками, платформами и инструментами. Код и логика не могут быть легко общими для клиента и сервера, что приводит к дублированию и проектированию. Также может быть сложно поддерживать экосистему JavaScript, которая ведет историю развития на скорости бреакнекк. Настройки интерфейсной платформы и средства сборки быстро меняются. В отрасли наблюдалось прогресс от grunt до gulp в пакете и т. д. Одно и то же обновление рестлесс было выполнено с интерфейсными платформами, такими как jQuery, выколачивание, угловой, реагирующий и Vue. Но, учитывая Монополи в браузере JavaScript, в важности было мало выбора. То есть, пока сообщество веб-сообщества не собралось и не послужило *Миракле* !

## <a name="webassembly-fulfills-a-need"></a>Сборка соответствия требованиям

В 2015 основные поставщики браузера присоединяются к группе сообщества W3C для создания нового открытого Web Standard, именуемого веб-сборкой. Веб-сборка — это байтовый код для Интернета. Если вы можете скомпилировать код в веб – сборку, он может запускаться в любом браузере на любой платформе с частотой почти в машинном коде. Начальные усилия на C/C++. Результат был очень значительным демонстрациями выполнения машинных графических модулей с трехмерной графикой непосредственно в браузере без подключаемых модулей. Так как эта сборка была стандартизована и реализована всеми основными браузерами.

Работа над запуском .NET для этой сборки была объявлена в конце 2017 и ожидается в 2020, включая поддержку из .NET 5. Возможность запуска кода .NET непосредственно в браузере обеспечивает веб-разработку с полным стеком на платформе .NET.

## <a name="blazor-full-stack-web-development-with-net"></a>Блазор: полноценная разработка веб-приложений с помощью .NET

В свою сторону, возможность запуска кода .NET в браузере не предоставляет полноценного интерфейса для создания клиентских веб-приложений. Именно здесь Блазор входит в. Blazor представляет собой клиентскую платформу веб-интерфейса, основанную на C#, а не JavaScript. Блазор может выполняться непосредственно в браузере через веб – сборку. Подключаемые модули браузера не требуются. Кроме того, приложения Блазор могут запускать на стороне сервера в .NET Core и выполнять все взаимодействия пользователя в режиме реального времени с помощью браузера.

В блазор есть отличная поддержка средств в Visual Studio и Visual Studio Code. Платформа также включает в себя полную модель компонентов пользовательского интерфейса и имеет встроенные средства для:

- Формы и проверка
- Внедрение зависимостей
- маршрутизация на стороне клиента;
- Макеты
- Отладка в браузере
- Взаимодействие с JavaScript

Блазор много общего с веб-формами ASP.NET. Обе платформы предлагают модели программирования пользовательского интерфейса с отслеживанием состояния на основе компонентов, управляемых событиями. Основное архитектурное различие заключается в том, что ASP.NET веб-формы выполняются только на сервере. Блазор может выполняться на клиенте в браузере. Но если вы поступаете от фона веб-форм ASP.NET, у вас есть множество Блазор, которые будут знакомы. Блазор — это естественное решение для разработчиков веб-форм ASP.NET, которые ищут способ использования преимуществ разработки на стороне клиента и возможностей .NET с открытым исходным кодом в различных платформах.

Эта книга содержит введение в Блазор, предназначенное специально для ASP.NET разработчиков веб-форм. Каждая концепция Блазор представлена в контексте аналогичных функций и методов веб-форм ASP.NET. К концу этой книги вы получите представление о следующих возможностях:

- Как создавать приложения Блазор.
- Как работает Блазор.
- Как Блазор относится к .NET Core.
- Разумные стратегии переноса существующих приложений ASP.NET Web Forms в Блазор, где это необходимо.
