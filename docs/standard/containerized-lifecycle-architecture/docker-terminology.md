# <a name="docker-terminology"></a><span data-ttu-id="2d4a4-103">Терминология Docker</span><span class="sxs-lookup"><span data-stu-id="2d4a4-103">Docker terminology</span></span>

<span data-ttu-id="2d4a4-104">В этом разделе приводятся термины и определения, необходимые при работе с инструментом Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-104">This section lists terms and definitions you should be familiar with before getting deeper into Docker.</span></span> <span data-ttu-id="2d4a4-105">Для определения, см. в разделе расширенной [Глоссарий](https://docs.docker.com/glossary/) предоставляемые Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-105">For further definitions, see the extensive [glossary](https://docs.docker.com/glossary/) provided by Docker.</span></span>

<span data-ttu-id="2d4a4-106">**Образ контейнера**: пакет со всеми зависимостями и сведениями, необходимыми для создания контейнера.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-106">**Container image**: A package with all the dependencies and information needed to create a container.</span></span> <span data-ttu-id="2d4a4-107">Образ включает в себя все зависимости (например, платформы), а также конфигурацию развертывания и выполнения для среды выполнения контейнера.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-107">An image includes all the dependencies (such as frameworks) plus deployment and execution configuration to be used by a container runtime.</span></span> <span data-ttu-id="2d4a4-108">Как правило, образ создается на основе нескольких базовых образов, наслоенных друг на друга в файловой системе контейнера.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-108">Usually, an image derives from multiple base images that are layers stacked on top of each other to form the container's filesystem.</span></span> <span data-ttu-id="2d4a4-109">После создания образ остается неизменным.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-109">An image is immutable once it has been created.</span></span>

<span data-ttu-id="2d4a4-110">**Dockerfile**: Текстовый файл, который содержит инструкции по созданию образа Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-110">**Dockerfile**: A text file that contains instructions on how to build a Docker image.</span></span> <span data-ttu-id="2d4a4-111">Это как пакетный скрипт, первая строка состояния в базовый образ начинаются с, а затем следуйте инструкциям, чтобы установить необходимые программы, копирование файлов и т. д., пока не дойдете рабочей среды необходимо.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-111">It's like a batch script, the first line states the base image to begin with and then follow the instructions to install required programs, copy files, and so on, until you get the working environment you need.</span></span>

<span data-ttu-id="2d4a4-112">**Сборка**: действие по созданию образа контейнера на основе сведений и контекста, предоставленных файлом Dockerfile, а также дополнительных файлов в папке, где создается образ.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-112">**Build**: The action of building a container image based on the information and context provided by its Dockerfile, plus additional files in the folder where the image is built.</span></span> <span data-ttu-id="2d4a4-113">Сборка образа с помощью Docker **`docker build`** команды.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-113">You can build images with the Docker **`docker build`** command.</span></span>

<span data-ttu-id="2d4a4-114">**Контейнер**: экземпляр образа Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-114">**Container**: An instance of a Docker image.</span></span> <span data-ttu-id="2d4a4-115">Контейнер отвечает за выполнение одного приложения, процесса или службы.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-115">A container represents the execution of a single application, process, or service.</span></span> <span data-ttu-id="2d4a4-116">Он состоит из содержимого образа Docker, среды выполнения и стандартного набора инструкций.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-116">It consists of the contents of a Docker image, an execution environment, and a standard set of instructions.</span></span> <span data-ttu-id="2d4a4-117">При масштабировании службы вы создаете несколько экземпляров контейнера из одного образа.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-117">When scaling a service, you create multiple instances of a container from the same image.</span></span> <span data-ttu-id="2d4a4-118">Или пакетное задание может создать несколько контейнеров из одного образа, передавая разные параметры каждому экземпляру.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-118">Or a batch job can create multiple containers from the same image, passing different parameters to each instance.</span></span>

<span data-ttu-id="2d4a4-119">**Тома**: предложите файловую систему с возможностью записи, которую может использовать этот контейнер.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-119">**Volumes**: Offer a writable filesystem that the container can use.</span></span> <span data-ttu-id="2d4a4-120">Поскольку образы доступны только для чтения, а большинству программ требуется возможность записи в файловую систему, тома добавляют слой с поддержкой записи поверх образа контейнера, который программы смогут использовать как файловую систему с возможностью записи.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-120">Since images are read-only but most programs need to write to the filesystem, volumes add a writable layer, on top of the container image, so the programs have access to a writable filesystem.</span></span> <span data-ttu-id="2d4a4-121">Программа не знает, обращается к многоуровневой файловую систему, это просто файловой системы, как обычно.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-121">The program doesn't know it's accessing a layered filesystem, it's just the filesystem as usual.</span></span> <span data-ttu-id="2d4a4-122">Тома размещаются в системе узла под управлением Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-122">Volumes live in the host system and are managed by Docker.</span></span>

<span data-ttu-id="2d4a4-123">**Тег**: метка для образа, которая позволяет различать разные образы или версии одного образа (в зависимости от номера версии или целевой среды).</span><span class="sxs-lookup"><span data-stu-id="2d4a4-123">**Tag**: A mark or label you can apply to images so that different images or versions of the same image (depending on the version number or the target environment) can be identified.</span></span>

<span data-ttu-id="2d4a4-124">**Многоэтапная сборка**: этот компонент входит в Docker, начиная с версии 17.05, и позволяет сократить итоговый размер образов.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-124">**Multi-stage Build**: Is a feature, since Docker 17.05 or higher, that helps to reduce the size of the final images.</span></span> <span data-ttu-id="2d4a4-125">Если попытаться кратко описать эту функцию, многоэтапная сборка позволяет использовать, крупный базовый образ с пакетом SDK для компиляции и публикации приложения, а затем создать итоговый образ намного меньшего размера с помощью небольшого базового образа для среды выполнения, размещенного в папке публикации.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-125">In a few sentences, with multi-stage build you can use, for example, a large base image, containing the SDK, for compiling and publishing the application and then using the publishing folder with a small runtime-only base image, to produce a much smaller final image</span></span>

<span data-ttu-id="2d4a4-126">**Репозиторий**: коллекция связанных образов Docker, помеченная тегом, указывающим на версию образа.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-126">**Repository (repo)**: A collection of related Docker images, labeled with a tag that indicates the image version.</span></span> <span data-ttu-id="2d4a4-127">Некоторые репозитории содержат несколько вариантов одного образа, например образ с пакетом средств разработки (больший объем), образ только со средой выполнения (меньший объем) и т. д. Эти варианты можно пометить тегами.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-127">Some repos contain multiple variants of a specific image, such as an image containing SDKs (heavier), an image containing only runtimes (lighter), etc. Those variants can be marked with tags.</span></span> <span data-ttu-id="2d4a4-128">Один репозиторий может содержать варианты платформ, например образ Linux и образ Windows.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-128">A single repo can contain platform variants, such as a Linux image and a Windows image.</span></span>

<span data-ttu-id="2d4a4-129">**Реестр**: служба, предоставляющая доступ к репозиториям.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-129">**Registry**: A service that provides access to repositories.</span></span> <span data-ttu-id="2d4a4-130">Реестр по умолчанию для большинства общедоступных образов — [Центр Docker](https://hub.docker.com/) (принадлежащий Docker как организации).</span><span class="sxs-lookup"><span data-stu-id="2d4a4-130">The default registry for most public images is [Docker Hub](https://hub.docker.com/) (owned by Docker as an organization).</span></span> <span data-ttu-id="2d4a4-131">Реестр обычно содержит репозитории нескольких команд.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-131">A registry usually contains repositories from multiple teams.</span></span> <span data-ttu-id="2d4a4-132">Компании часто используют частные реестры для хранения своих образов и управления ими.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-132">Companies often have private registries to store and manage images they've created.</span></span> <span data-ttu-id="2d4a4-133">Еще один пример — реестр контейнеров Azure.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-133">Azure Container Registry is another example.</span></span>

<span data-ttu-id="2d4a4-134">**Мультиархитектурный образ**: Для нескольких архитектуры — это функция, упрощает выбор соответствующий образ в соответствии с платформой, на котором Docker выполняется, например, когда Dockerfile запрашивает базовый образ **`FROM mcr.microsoft.com/dotnet/core/sdk:2.2`** из реестра фактически возвращает **`2.2-nanoserver-1709`**, **`2.2-nanoserver-1803`**, **`2.2-nanoserver-1809`** или **`2.2-stretch`**, в зависимости от операционной системы и версии, где выполняется Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-134">**Multi-arch image**: For multi-architecture, is a feature that simplifies the selection of the appropriate image, according to the platform where Docker is running, for example, when a Dockerfile requests a base image **`FROM mcr.microsoft.com/dotnet/core/sdk:2.2`** from the registry it actually gets **`2.2-nanoserver-1709`**, **`2.2-nanoserver-1803`**, **`2.2-nanoserver-1809`** or **`2.2-stretch`**, depending on the operating system and version where Docker is running.</span></span>

<span data-ttu-id="2d4a4-135">**Центр Docker**: общедоступный реестр для загрузки образов и работы с ними.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-135">**Docker Hub**: A public registry to upload images and work with them.</span></span> <span data-ttu-id="2d4a4-136">Центр Docker обеспечивает размещение образов Docker и интеграцию с GitHub и Bitbucket, предоставляет общедоступные или частные реестры, триггеры сборки и веб-перехватчики.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-136">Docker Hub provides Docker image hosting, public or private registries, build triggers and web hooks, and integration with GitHub and Bitbucket.</span></span>

<span data-ttu-id="2d4a4-137">**Реестр контейнеров Azure**: общедоступный ресурс для работы с образами Docker и их компонентами в Azure.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-137">**Azure Container Registry**: A public resource for working with Docker images and its components in Azure.</span></span> <span data-ttu-id="2d4a4-138">Он предоставляет реестр, близок к вашим развертываниям в Azure и который позволяет контролировать доступ, что позволяет использовать группы Azure Active Directory и разрешения.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-138">This provides a registry that's close to your deployments in Azure and that gives you control over access, making it possible to use your Azure Active Directory groups and permissions.</span></span>

<span data-ttu-id="2d4a4-139">**Доверенный реестр Docker (DTR)**: служба реестра Docker (из Docker), которую можно установить локально, чтобы она находилась в центре данных и сети организации.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-139">**Docker Trusted Registry (DTR)**: A Docker registry service (from Docker) that can be installed on-premises so it lives within the organization's datacenter and network.</span></span> <span data-ttu-id="2d4a4-140">Это удобно для частных образов, которыми необходимо управлять внутри предприятия.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-140">It's convenient for private images that should be managed within the enterprise.</span></span> <span data-ttu-id="2d4a4-141">Доверенный реестр Docker входит в Центр данных Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-141">Docker Trusted Registry is included as part of the Docker Datacenter product.</span></span> <span data-ttu-id="2d4a4-142">Дополнительные сведения см. в разделе [Доверенный реестр Docker (DTR)](https://docs.docker.com/docker-trusted-registry/overview/).</span><span class="sxs-lookup"><span data-stu-id="2d4a4-142">For more information, see [Docker Trusted Registry (DTR)](https://docs.docker.com/docker-trusted-registry/overview/).</span></span>

<span data-ttu-id="2d4a4-143">**Docker Community Edition (CE)**: инструменты разработки в среде Windows и macOS для локальной сборки, выполнения и тестирования контейнеров.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-143">**Docker Community Edition (CE)**: Development tools for Windows and macOS for building, running, and testing containers locally.</span></span> <span data-ttu-id="2d4a4-144">Docker CE для Windows предоставляет среды разработки для контейнеров Linux и Windows.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-144">Docker CE for Windows provides development environments for both Linux and Windows Containers.</span></span> <span data-ttu-id="2d4a4-145">Узел Linux Docker на Windows базируется на виртуальной машине [Hyper-V](https://www.microsoft.com/cloud-platform/server-virtualization).</span><span class="sxs-lookup"><span data-stu-id="2d4a4-145">The Linux Docker host on Windows is based on a [Hyper-V](https://www.microsoft.com/cloud-platform/server-virtualization) virtual machine.</span></span> <span data-ttu-id="2d4a4-146">Узел для контейнеров Windows базируется непосредственно на Windows.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-146">The host for Windows Containers is directly based on Windows.</span></span> <span data-ttu-id="2d4a4-147">Docker CE для Mac базируется на платформе Apple Hypervisor и [гипервизоре xhyve](https://github.com/mist64/xhyve), который предоставляет узлу Linux Docker виртуальную машину в Mac OS X. Docker CE для Windows и для Mac заменил Docker Toolbox на базе Oracle VirtualBox.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-147">Docker CE for Mac is based on the Apple Hypervisor framework and the [xhyve hypervisor](https://github.com/mist64/xhyve), which provides a Linux Docker host virtual machine on Mac OS X. Docker CE for Windows and for Mac replaces Docker Toolbox, which was based on Oracle VirtualBox.</span></span>

<span data-ttu-id="2d4a4-148">**Docker Enterprise Edition (EE)**: корпоративная версия инструментов Docker для разработки в среде Windows и Linux.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-148">**Docker Enterprise Edition (EE)**: An enterprise-scale version of Docker tools for Linux and Windows development.</span></span>

<span data-ttu-id="2d4a4-149">**Compose**: программа командной строки и формат файлов YAML с метаданными для определения и выполнения многоконтейнерных приложений.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-149">**Compose**: A command-line tool and YAML file format with metadata for defining and running multi-container applications.</span></span> <span data-ttu-id="2d4a4-150">Вы определяете одно приложение на основе нескольких образов с помощью одного или нескольких файлов .yml, которые могут изменять значения в зависимости от среды.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-150">You define a single application based on multiple images with one or more .yml files that can override values depending on the environment.</span></span> <span data-ttu-id="2d4a4-151">После создания определения, можно развернуть все многоконтейнерное приложение с помощью одной команды (docker-compose up), создает один контейнер на образ на узле Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-151">After you've created the definitions, you can deploy the whole multi-container application with a single command (docker-compose up) that creates a container per image on the Docker host.</span></span>

<span data-ttu-id="2d4a4-152">**Кластер**: коллекция узлов Docker, представленная в виде единого виртуального узла Docker, чтобы можно было масштабировать приложение в нескольких экземплярах служб, распределенных по нескольким узлам кластера.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-152">**Cluster**: A collection of Docker hosts exposed as if it were a single virtual Docker host, so that the application can scale to multiple instances of the services spread across multiple hosts within the cluster.</span></span> <span data-ttu-id="2d4a4-153">Кластеры Docker можно создавать с помощью Kubernetes, Azure Service Fabric, Docker Swarm и (или) Mesosphere DC/OS.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-153">Docker clusters can be created with Kubernetes, Azure Service Fabric, Docker Swarm and Mesosphere DC/OS.</span></span>

<span data-ttu-id="2d4a4-154">**Оркестратор**: инструмент, упрощающий управление кластерами и узлами Docker.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-154">**Orchestrator**: A tool that simplifies management of clusters and Docker hosts.</span></span> <span data-ttu-id="2d4a4-155">Оркестраторы позволяют управлять их образы, контейнерами и узлами через интерфейс командной строки (CLI) или графический Интерфейс.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-155">Orchestrators enable you to manage their images, containers, and hosts through a command-line interface (CLI) or a graphical UI.</span></span> <span data-ttu-id="2d4a4-156">Вы можете управлять соединениями контейнеров, конфигурациями, балансировкой нагрузки, обнаружением служб, высоким уровнем доступности, конфигурацией узлов Docker и многим другим.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-156">You can manage container networking, configurations, load balancing, service discovery, high availability, Docker host configuration, and more.</span></span> <span data-ttu-id="2d4a4-157">Оркестратор используется для выполнения, распределения, масштабирования и восстановления рабочих нагрузок в коллекции узлов.</span><span class="sxs-lookup"><span data-stu-id="2d4a4-157">An orchestrator is responsible for running, distributing, scaling, and healing workloads across a collection of nodes.</span></span> <span data-ttu-id="2d4a4-158">В качестве оркестраторов обычно используются те же продукты, которые обеспечивают кластерную инфраструктуру, например Kubernetes и Azure Service Fabric (на рынке доступны и другие предложения).</span><span class="sxs-lookup"><span data-stu-id="2d4a4-158">Typically, orchestrator products are the same products that provide cluster infrastructure, like Kubernetes and Azure Service Fabric, among other offerings in the market.</span></span>
