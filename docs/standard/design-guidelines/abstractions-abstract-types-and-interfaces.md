# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="49b92-102">Абстракции (абстрактные типы и интерфейсы)</span><span class="sxs-lookup"><span data-stu-id="49b92-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="49b92-103">Абстракция — это тип, который описывает контракт, но не содержит полной реализации контракта.</span><span class="sxs-lookup"><span data-stu-id="49b92-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="49b92-104">Абстракции обычно реализуются как абстрактных классов или интерфейсов, и они поставляются с четко определенный набор справочная документация, описывающие необходимую семантику типов, реализующих контракт.</span><span class="sxs-lookup"><span data-stu-id="49b92-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="49b92-105">Ниже перечислены некоторые из важнейших абстракций в .NET Framework <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, и <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="49b92-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>  
  
 <span data-ttu-id="49b92-106">Платформы можно расширить путем реализации конкретный тип, поддерживает контракт абстракции и использовать этот конкретный тип framework API-интерфейсы много (работе) абстракции.</span><span class="sxs-lookup"><span data-stu-id="49b92-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>  
  
 <span data-ttu-id="49b92-107">Значимые и полезные абстракции, который способен выдержать испытание времени для разработки очень сложно.</span><span class="sxs-lookup"><span data-stu-id="49b92-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="49b92-108">Основные сложности является получение правильного набора элементов, не больше и не меньше.</span><span class="sxs-lookup"><span data-stu-id="49b92-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="49b92-109">Если это абстракция имеет слишком много элементов, становится трудно или даже невозможно реализовать.</span><span class="sxs-lookup"><span data-stu-id="49b92-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="49b92-110">Если он имеет слишком мало элементов для обещанный функциональных возможностей, он не стал бесполезен в много интересных ситуаций.</span><span class="sxs-lookup"><span data-stu-id="49b92-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>  
  
 <span data-ttu-id="49b92-111">Слишком много абстракции в платформе также отрицательно повлиять на удобство использования платформы.</span><span class="sxs-lookup"><span data-stu-id="49b92-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="49b92-112">Часто бывает довольно сложно понять абстракцию, не понимая, как ИТ вписывается в общую картину конкретные реализации и API-интерфейсы, работающие с абстракцией.</span><span class="sxs-lookup"><span data-stu-id="49b92-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="49b92-113">Кроме того имена абстракций и члены являются обязательно абстрактными, часто делает их интерпретировать и unapproachable без первого основные сведения о более широком контексте их использования.</span><span class="sxs-lookup"><span data-stu-id="49b92-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>  
  
 <span data-ttu-id="49b92-114">Тем не менее абстракции предоставляют очень эффективно расширяемости, часто не может совпадать с другие механизмы расширяемости.</span><span class="sxs-lookup"><span data-stu-id="49b92-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="49b92-115">Они являются основой множество архитектурных шаблонов, таких как подключаемые модули, инверсии управления (IoC), конвейеры и т. д.</span><span class="sxs-lookup"><span data-stu-id="49b92-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="49b92-116">Они также крайне важно для тестирования платформ.</span><span class="sxs-lookup"><span data-stu-id="49b92-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="49b92-117">Хороший абстракции позволяют заглушки для интенсивной зависимости для целей модульного тестирования.</span><span class="sxs-lookup"><span data-stu-id="49b92-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="49b92-118">Таким образом абстракции отвечают за популярных разнообразие современных платформ объектно ориентированного.</span><span class="sxs-lookup"><span data-stu-id="49b92-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>  
  
 <span data-ttu-id="49b92-119">**X DO NOT** предоставляют абстракции, пока не проверите их путем разработки нескольких конкретных реализаций и использует эти абстракции API-интерфейсы.</span><span class="sxs-lookup"><span data-stu-id="49b92-119">**X DO NOT** provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>  
  
 <span data-ttu-id="49b92-120">**✓ DO** тщательно выбирайте между абстрактным классом и интерфейсом при проектировании абстракции.</span><span class="sxs-lookup"><span data-stu-id="49b92-120">**✓ DO** choose carefully between an abstract class and an interface when designing an abstraction.</span></span>  
  
 <span data-ttu-id="49b92-121">**✓ CONSIDER** предоставление тесты конкретных реализаций абстракций.</span><span class="sxs-lookup"><span data-stu-id="49b92-121">**✓ CONSIDER** providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="49b92-122">Такие тесты позволят пользователям проверить их реализации правильной реализации контракта.</span><span class="sxs-lookup"><span data-stu-id="49b92-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
