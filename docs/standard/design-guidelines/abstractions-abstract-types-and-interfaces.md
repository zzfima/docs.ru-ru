# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="04443-102">Абстракции (абстрактные типы и интерфейсы)</span><span class="sxs-lookup"><span data-stu-id="04443-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="04443-103">Абстракция — это тип, описывающий контракт, но не предоставляющий полную реализацию контракта.</span><span class="sxs-lookup"><span data-stu-id="04443-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="04443-104">Абстракции обычно реализуются как абстрактные классы или интерфейсы, и они поставляются с четко определенным набором справочной документации, описывающей необходимую семантику типов, реализующих контракт.</span><span class="sxs-lookup"><span data-stu-id="04443-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="04443-105">Некоторые из наиболее важных абстракций в .NET Framework включают <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>и <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="04443-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>  
  
 <span data-ttu-id="04443-106">Платформы можно расширять, реализуя конкретный тип, который поддерживает контракт абстракции и использует этот конкретный тип с API-интерфейсами платформы, которые потребляют (работает на) абстракцию.</span><span class="sxs-lookup"><span data-stu-id="04443-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>  
  
 <span data-ttu-id="04443-107">Осмысленная и полезная абстракция, способная выдержать тестирование времени, очень сложно спроектировать.</span><span class="sxs-lookup"><span data-stu-id="04443-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="04443-108">Основной трудностью является получение правильного набора членов, но не меньше.</span><span class="sxs-lookup"><span data-stu-id="04443-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="04443-109">Если у абстракции слишком много членов, то реализовать ее сложно или даже невозможно.</span><span class="sxs-lookup"><span data-stu-id="04443-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="04443-110">Если у вас слишком мало членов для функции обещания, она становится бесполезными во многих интересных сценариях.</span><span class="sxs-lookup"><span data-stu-id="04443-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>  
  
 <span data-ttu-id="04443-111">Слишком много абстракций в платформе также негативно влияют на удобство использования платформы.</span><span class="sxs-lookup"><span data-stu-id="04443-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="04443-112">Часто бывает довольно трудно понять абстракцию, не зная того, как она помещается в более крупную картину конкретных реализаций и интерфейсов API, работающих над абстракцией.</span><span class="sxs-lookup"><span data-stu-id="04443-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="04443-113">Кроме того, имена абстракций и их членов обязательно являются абстрактными, что часто делает их непонятными и неоправданными без предварительного понимания более широкого контекста их использования.</span><span class="sxs-lookup"><span data-stu-id="04443-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>  
  
 <span data-ttu-id="04443-114">Однако абстракции предоставляют чрезвычайно мощную расширяемость, которую другие механизмы расширяемости часто не могут сопоставлять.</span><span class="sxs-lookup"><span data-stu-id="04443-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="04443-115">Они являются основой многих архитектурных шаблонов, таких как подключаемые модули, инверсия управления (IoC), конвейеры и т. д.</span><span class="sxs-lookup"><span data-stu-id="04443-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="04443-116">Они также чрезвычайно важны для тестирования платформ.</span><span class="sxs-lookup"><span data-stu-id="04443-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="04443-117">Хорошая абстракция позволяет создать заглушки для модульного тестирования.</span><span class="sxs-lookup"><span data-stu-id="04443-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="04443-118">В целом, абстракции отвечают за насыщенность современных объектно-ориентированных платформ.</span><span class="sxs-lookup"><span data-stu-id="04443-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>  
  
 <span data-ttu-id="04443-119">**X DO NOT** предоставляют абстракции, пока не проверите их путем разработки нескольких конкретных реализаций и использует эти абстракции API-интерфейсы.</span><span class="sxs-lookup"><span data-stu-id="04443-119">**X DO NOT** provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>  
  
 <span data-ttu-id="04443-120">**✓ DO** тщательно выбирайте между абстрактным классом и интерфейсом при проектировании абстракции.</span><span class="sxs-lookup"><span data-stu-id="04443-120">**✓ DO** choose carefully between an abstract class and an interface when designing an abstraction.</span></span>  
  
 <span data-ttu-id="04443-121">**✓ CONSIDER** предоставление тесты конкретных реализаций абстракций.</span><span class="sxs-lookup"><span data-stu-id="04443-121">**✓ CONSIDER** providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="04443-122">Такие тесты должны дать пользователям возможность проверить, правильно ли реализуются их реализации.</span><span class="sxs-lookup"><span data-stu-id="04443-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
